#  页式管理基础

> [!note] 
>
> + 程序：静止的，存放在磁盘上的可执行文件
> + 进程：动态的，进程包括程序和程序处理对象（数据集），是一个程序的执行过程，是分配资源的基本单位。通常把进程分为系统进程和用户进程
> + 作业：用户需要计算机完成的某项任务，是要求计算机所作工作的集合
> + 作业 = 程序 + 数据 + 操作说明书（通常来说）
> + 进程 = 进程控制块PCB + 程序 + 数据集合

**页式管理的基本思想：**把一个逻辑地址连续的程序分散存放到若干不连续的内存区域内，并保证程序的正确执行

**优点：** 充分利用内存，减少移动开销

#### 纯分页系统（基本分页）

> 不具备页面对换功能，不支持虚拟存储器的分页存储管理方式

**工作过程**：在调度一个作业时，必须把它的所有页一次装到主存的页框内；如果当时页框数不足，则该作业必须等待，系统再调度另外作业。

**优点：**

+ 无外碎片
+ 一个程序不必连续存放
+ 便于改变程序占用空间的大小

**缺点：** 程序全部装入内存，有内碎片

以下介绍页式管理的基本方式：

> [!note]
>
> + 页（页面page）：逻辑地址空间上大小相等的片
> + 页框（存储块）：物理内存的存储空间上分出的与页面大小相同的片
> + 页表：为便于在内存找到进程每个页面对应的页框，分页系统为每个进程配置一张页表
> + 页表项：含有页框号、脏位 (dirty)、有效位 (valid)、引用 (reference) 等等信息，一般 4 字节
> + 页表寄存器：记录页表的基址和长度

对于 $m$ 位地址空间，页大小为 $2^k$ byte，则逻辑地址可分为：

| 页号：高m-k位 | 页内偏移：低k位 |
| ------------- | --------------- |

根据页号，可从页表读出对应的项，项中记录着对应的页框号，将页框号替换掉页号，得到物理地址

页面大小的分析：

|          | 大页面                             | 小页面                             |
| -------- | ---------------------------------- | ---------------------------------- |
| 碎片     | 内碎片多，内存利用率较低           | 碎片少，内存利用率较高             |
| 占用内存 | 页面数少，页表长度减少，占用内存小 | 页面数多，页表长度增加，占用内存多 |
| miss发生     | miss 少                 | miss 多                    |

访问一个数据需要访问内存 **2次**（页表一次，内存一次）

#### 多级页表

一级页表的问题：若逻辑地址空间大，则划分的页就较多，页表就大，占用存储空间大（要求连续）

**解决方案：** 将页表再进行分页，离散地将各个页表页面存放在不同的物理块中，再建立一个外层页表（页目录）用来记录页表页面对应的物理块

反过来说，可以得到多级页表的**优点：**

+ 只为实际使用的页表分配页，减少空间开销
+ 让单个页表占用空间减小，减少对大片连续存储空间的需求，有利于减少碎片
+ 便于页表共享与隔离

例如，$m$ 位地址空间，页大小为$2^k Byte$ ，每个页表项大小为$2^nByte$，则应为： 

| 页表号：m+n-2k | 页号：k-n | 页内偏移：低k位 |
| -------------- | --------- | --------------- |

> [!NOTE]
>
> $k-n$ 是由于页表需要连续空间来存放页表项，如果超过一页则会降低调度效率，如果不足一页则会浪费空间（产生内碎片），因此恰好是一页比较合适（一个页表页大小为 $2^k$ Byte、每项 $2^n$ Byte，所以一页能放 $2^(k-n)$ 项）

> [!CAUTION]
>
> 两级页表与单级页表相比，页表占用的空间总量上限其实变大了，因为页目录项需要多占空间，但是通常来说，只有使用的页表需要调入内存，瞬时占用空间变小了

#### 快表（TLB，联想存储器）

> 快表即为页表的高速缓冲存储器（Cache）

有效内存访问时间（$t$ 为查找TLB所需时间，$T$ 为访问内存时间，$\alpha$ 为平均命中率）： $(1-\alpha)(t+T+T)+\alpha (t+T)$

**替换策略：** LRU，$\dots$

#### 哈希页表

在讲哈希页表前，先要讲反置页表，讲反置页表前，要先说说普通页表的问题

普通页表按虚拟地址索引，页表项存放物理地址和权限位，页表项个数上限取决于虚拟地址空间大小，这种方式虽然比较自然（符合正常映射的想法），但当**虚拟地址空间大于物理地址空间**（这通常会发生）时，可能会出现内存都用来装页表项了，浪费大量物理内存

因此提出**反置页表：按页框号排列，页表项存放页号和占用的进程pid**，这种页表只与物理内存大小相关

但是，反置页表违背了传统映射的思路，因此查找困难。为解决这个问题，引入了哈希页表快速查找对应的页框号

不过使用反置页表的内存系统很难处理内存共享，即一个页框只能由一个进程控制

#### 页共享与保护

**保护：**

+ 地址越界保护
+ 在页表中设置保护位（定义操作权限：只读，读写，执行等）

**共享带来的问题：**共享数据和不共享数据被划分到一个页里，不易保密

段式管理是一个比较好的解决这个问题的方式。
