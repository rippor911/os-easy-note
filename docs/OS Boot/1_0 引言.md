# 引言

## 一、这一章要解决什么问题

操作系统不是一个孤立的软件包，而是位于用户程序和硬件之间的系统层。学习 OS Boot 章节时，先要回答三个问题：

+ **What：** 什么是操作系统，它向上提供什么接口，向下管理什么资源。
+ **Why：** 为什么普通程序不应该直接控制全部硬件。
+ **How：** 一个程序如何借助操作系统真正运行起来，操作系统自己又如何从上电状态开始运行。

一个简单的 `hello world` 程序背后，其实已经包含了进程创建、程序装入、地址空间、缺页异常、系统调用、终端设备和显示输出等内容。后面的启动过程也是同一个问题的另一面：控制权如何从硬件和固件逐步交给内核。

## 二、什么是操作系统

操作系统可以非严格地理解为：**一组管理计算机硬件资源的软件集合，它向计算机程序提供共性的服务**。

从不同角度看，操作系统至少承担三类角色：

| 角度 | 关注点 | 说明 |
| --- | --- | --- |
| 用户和程序的接口 | UI、API、ABI、ISA | 用户和应用程序不直接面对复杂硬件，而是通过命令、图形界面、系统调用和二进制接口使用系统能力 |
| 资源管理者 | CPU、内存、I/O、文件 | 分配、回收、保护和调度系统资源，避免多个程序互相破坏 |
| 扩充机或虚拟机 | 抽象 | 把复杂硬件抽象成文件、进程、虚拟地址空间、设备等更容易使用的对象 |

> [!NOTE]
> 操作系统的重点不是“替用户操作电脑”，而是**在多个程序共享硬件时，提供可控、可保护、可复用的运行环境**。

## 三、自顶向下与自底向上

**自顶向下看：** 操作系统屏蔽硬件复杂性，向程序提供抽象。

+ 文件系统让程序不必直接关心磁盘柱面、磁头、扇区。
+ 虚拟内存让程序先使用连续的逻辑地址空间，再由硬件和内核完成映射。
+ 进程让程序拥有相对独立的执行上下文和资源视图。

**自底向上看：** 操作系统负责复用和协调有限硬件资源。

+ 在时间维度上复用 CPU，例如中断、时间片、调度。
+ 在空间维度上复用内存和外存，例如地址空间、分页、文件系统。
+ 在设备维度上复用 I/O，例如缓冲、假脱机和设备驱动。

因此，操作系统设计目标通常包括：

+ **方便性：** 让更多人能更容易使用计算机。
+ **有效性：** 提高资源利用率和系统吞吐量。
+ **安全性：** 限制程序权限，防止用户程序破坏内核或其他程序。
+ **可扩展性：** 在硬件、应用和负载变化时仍能演进。

## 四、发展脉络

操作系统的很多机制都来自硬件能力和使用场景的变化。

| 阶段 | 典型系统形态 | 主要问题 | 对 OS 的影响 |
| --- | --- | --- | --- |
| 早期手工操作 | 人工装入和运行程序 | 机器昂贵，人工切换低效 | 需要自动控制作业执行 |
| 监控程序与批处理 | 联机、脱机批处理 | CPU 与 I/O 速度差异大 | 引入作业队列、批量处理、假脱机 |
| 多道程序 | 多个程序同时驻留内存 | CPU、内存、I/O 共享 | 需要中断、调度、保护和内存管理 |
| 分时系统 | 多用户交互 | 响应时间和公平性 | 时间片、终端、系统调用、虚拟内存成为核心机制 |
| PC、分布式、嵌入式 | 个人计算、网络、实时场景 | 硬件和需求多样化 | 出现不同结构和不同目标的操作系统 |

分时系统是一个关键分水岭。它把“一个程序独占所有硬件”改成“多个程序共享硬件”，因此应用程序不能再随意访问全部硬件能力，必须通过内核、系统调用、页式内存管理等机制受控地使用资源。

## 五、基本特征

操作系统通常具有四个基本特征：

+ **并发：** 多个程序在时间上重叠推进。并发不等于并行，并行强调同一时刻在多个处理器上同时执行。
+ **共享：** 多个进程共享系统资源，包括互斥共享和宏观上的同时访问。
+ **虚拟：** 把一个物理资源表现为多个逻辑资源，例如虚拟 CPU、虚拟内存、SPOOLing 设备。
+ **异步性：** 程序执行经常“走走停停”，可能因为 I/O、调度、缺页、锁等待等原因暂停和恢复。

> [!CAUTION]
> “虚拟”不是“假的”。虚拟化的重点是通过映射、调度和保护，让程序看到一个更规则、更容易使用的资源视图。

## 六、硬件基础与基本机制

操作系统建立在硬件机制之上。CPU、内存和 I/O 设备的速度差异很大，计算速度通常远快于访存和 I/O。为了提高整体效率，操作系统必须让 CPU 在等待慢设备时去执行其他工作，并通过缓存、缓冲、中断和调度降低等待成本。

几个基本机制需要先区分清楚：

| 机制 | 触发原因 | 同步性 | 返回行为 | 例子 |
| --- | --- | --- | --- | --- |
| 中断 interrupt | 外部设备或时钟发出信号 | 异步 | 通常返回下一条指令 | 定时器中断、I/O 完成中断 |
| 陷阱 trap | 程序有意执行特殊指令 | 同步 | 通常返回下一条指令 | 系统调用 |
| 故障 fault | 当前指令执行遇到可恢复问题 | 同步 | 修复后重试当前指令 | 缺页异常、部分保护异常 |
| 终止 abort | 不可恢复错误 | 同步或异步 | 通常不返回 | 严重硬件错误 |

系统调用可以看作一种受控陷入内核的方式。用户程序通过特定指令进入内核态，CPU 切换到内核栈，根据中断向量或异常入口执行内核处理程序，处理结束后再通过返回指令回到用户态。

## 七、功能与结构

操作系统的功能可以按资源类型理解：

| 功能 | 主要内容 |
| --- | --- |
| 处理机管理 | 进程或线程控制、同步、通信、调度 |
| 存储器管理 | 内存分配与回收、地址映射、内存保护、内存扩充 |
| 设备管理 | 缓冲管理、设备分配、设备处理、中断处理 |
| 文件管理 | 文件存取、目录管理、共享和保护 |
| 作业控制 | 作业提交、运行控制、用户命令或脚本接口 |

操作系统结构则体现“复杂系统如何组织”。常见思路包括：

+ **模块接口法：** 按功能划分模块，但要避免接口混乱和循环依赖。
+ **有序分层法：** 上层调用下层，通过层次减少任意依赖。
+ **虚拟机结构：** 在裸机之上提供多个相互隔离的虚拟机器，保护性好但有性能开销。
+ **微内核结构：** 内核只保留中断处理、IPC、基本调度等核心能力，文件系统、网络、设备管理等放到用户态服务中。
+ **机制与策略分离：** 内核提供“能做什么”的机制，具体“怎么选择”尽量由策略决定。

## 八、复习抓手

+ 操作系统的本质是**抽象 + 资源管理 + 保护**。
+ 多道程序和分时系统推动了中断、系统调用、内核态、虚拟内存等机制的发展。
+ 系统调用是同步陷阱，中断通常是异步事件，缺页异常是可恢复故障。
+ 后续章节中的进程、内存、文件和设备管理，本质上都是在解决“多个程序共享有限硬件”的问题。
