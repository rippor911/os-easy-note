# Boot 与启动

## 一、Boot 的核心矛盾

Bootstrapping 简称 boot，直观含义是“自己把自己拉起来”。计算机启动确实有类似矛盾：

+ 必须先运行一段程序，计算机才能进入可用状态。
+ 但程序又必须运行在已经被正确初始化的硬件环境上。

因此，启动前硬件只能处在一个**最安全、最通用、功能最弱**的初始状态。启动过程的本质，是逐步设置硬件、加载更强的程序、交接控制权，最终把系统带到可以运行操作系统和用户程序的状态。

> [!NOTE]
> OS 启动可以理解为一个“逐步释放系统灵活性”的过程：从固定入口、固定地址和极少功能开始，逐步获得内存、缓存、文件系统、设备驱动和多任务能力。

## 二、Bootloader 的职责

Bootloader 是操作系统内核运行之前的一段引导程序。它的名字可以拆成两层含义：

+ **Booter：** 初始化必要硬件，让系统至少部分运行起来。
+ **Loader：** 把操作系统映像加载到内存，并跳转到内核入口。

在嵌入式系统中，Bootloader 往往非常依赖具体硬件。即使 CPU 相同，不同开发板的 Flash、串口、内存控制器和外设也可能不同，所以 Bootloader 通常需要移植。MIPS 或其他嵌入式场景常见 U-Boot；传统 x86 Linux 场景常见 LILO 和 GRUB。

多数 Bootloader 可以分为两阶段：

| 阶段 | 典型实现 | 主要职责 |
| --- | --- | --- |
| stage1 | 汇编，强依赖 CPU 和板级硬件 | 固定入口启动、建立最小运行环境、初始化关键寄存器和早期内存 |
| stage2 | C 语言为主 | 提供较复杂功能，如串口输出、环境变量、存储设备、网络、文件系统、内核加载 |

## 三、MIPS 启动中的地址空间

MIPS 上电启动时，OS 尚未接管系统，TLB、Cache 和普通内存环境都不能假设已经可用。理解 MIPS 启动首先要看地址空间：

| 区域 | 地址范围 | 特性 | 启动阶段意义 |
| --- | --- | --- | --- |
| `kuseg` | `0x00000000` 到 `0x7fffffff` | 用户态可用，通常需要 MMU/TLB 转换 | MMU 未设置前不能依赖 |
| `kseg0` | `0x80000000` 到 `0x9fffffff` | 内核态，直接映射低 512 MiB 物理地址，通常经过 Cache | Cache 未初始化前不能随意使用 |
| `kseg1` | `0xa0000000` 到 `0xbfffffff` | 内核态，直接映射低 512 MiB 物理地址，不经过 Cache | 重启时最可靠的地址区域 |
| `kseg2/kseg3` | `0xc0000000` 到 `0xffffffff` | 内核态，通常需要 MMU/TLB 转换 | MMU 未设置前不应访问 |

MIPS 的启动入口地址是 `0xbfc00000`，它位于 `kseg1`。把高三位清零后，对应物理地址 `0x1fc00000`。选择这个区域的原因不是它更快，而是它在 TLB 和 Cache 都尚未准备好时仍能可靠工作。

## 四、MIPS / U-Boot 启动流程

U-Boot 的早期启动通常从 `start.S` 的 `_start` 开始：

1. 初始化中断向量。
2. 清零通用寄存器和部分协处理器寄存器。
3. 配置 `CP0_STATUS`，设置协处理器、中断和 CPU 运行级别。
4. 配置 `gp` 等基础寄存器。
5. 进入 `lowlevel_init.S`，配置 CPU 主频、总线、DDR 等低层参数。
6. 调用 cache 相关代码初始化 Cache。

早期阶段有一个特殊问题：DDR RAM 尚未配置好，但调用 C 函数需要栈。课件中的处理方式是先把一部分 Cache 当作临时 RAM 使用，把它配置成栈并锁定。这样访问栈地址时只会访问 Cache，不会访问尚不可用的真实 RAM。

进入 C 阶段后，`board_init_f` 会做一系列板级初始化：

+ `timer_init`：时钟初始化。
+ `env_init`：环境变量初始化。
+ `init_baudrate`、`serial_init`：串口速率和串口初始化。
+ `console_init_f`：控制台初始化。
+ `display_banner`：显示 U-Boot 版本和启动信息。
+ `init_func_ram`：初始化内存和 DDR 控制器。

当串口和内存可用后，U-Boot 会划分内存、初始化堆栈、为自身代码预留空间，并把代码从 Flash 搬到 RAM 中继续执行。后续 `board_init_r` 初始化 Flash、PCI、网口等外设，最后进入命令行或直接启动 Linux kernel。

## 五、Linux 内核接管控制权

Bootloader 的最终目标是把内核映像放到内存中的合适位置，然后跳转到内核入口。以 MIPS Linux 为例：

+ Bootloader 将 Linux 内核映像拷贝到 RAM，并移动到指定物理地址。
+ 内核入口位于体系结构相关的汇编代码中，例如 `/arch/mips/kernel/head.S` 的 `kernel_entry()`。
+ 入口代码初始化内核栈，清零未初始化数据段，为创建第一个进程准备环境。
+ 最后跳转到 `/init/main.c` 中的 `start_kernel()`。

`start_kernel()` 是 Linux 内核建立自身运行环境的主线，典型工作包括：

+ 设置 CPU ID，为多核环境做准备。
+ 初始化内核核心数据结构。
+ 关闭中断，避免初始化期间被异步事件打断。
+ 初始化体系结构相关的内存、页表和命令行参数。
+ 初始化调度器、定时器、软中断、时间系统和控制台。
+ 初始化 `vmalloc`、伙伴系统、内存管理等模块。
+ 在 `rest_init()` 中创建 `kernel_init` 和 `kthreadd`，并把当前执行流转为 idle 线程。
+ 通过调度进入多任务运行状态，由 `kernel_init` 启动第一个用户进程。

课程实验中的 MIPS 虚拟机和教学型操作系统比真实 Linux 简化很多，但边界一致：**Bootloader 准备环境并跳到内核，内核再建立自己的运行时和第一个用户态入口**。

## 六、x86 BIOS / MBR 启动链路

传统 x86 启动链路可以概括为：

1. 加电。
2. CPU 跳转到 BIOS 的固定物理地址，例如 Intel 80386 的 `0xffff0`。
3. BIOS 执行 POST 硬件自检。
4. BIOS 按启动顺序寻找可启动设备。
5. BIOS 从 MBR 加载启动扇区。
6. BIOS 把控制权交给 OS Bootloader。

BIOS 存放在主板 ROM 或 EEPROM 中，提供最底层的硬件设置和控制。它会读取 CPU、内存、磁盘、时钟、PnP 和启动顺序等信息，并在硬件自检通过后继续启动。

BIOS 的典型限制包括 16 位实模式、寻址能力有限、结构和移植性较差，以及长期兼容历史硬件和软件的成本。UEFI 试图解决这些问题：

| 对比项 | BIOS | UEFI |
| --- | --- | --- |
| 代码模式 | 传统 16 位实模式为主 | 可运行在 32 位或 64 位模式 |
| 结构 | 固化、整体性强 | 模块化，接口更清晰 |
| 分区支持 | 典型搭配 MBR | 支持 GPT，可从超过 2 TiB 的磁盘引导 |
| 扩展能力 | 受历史兼容限制较多 | 支持更灵活的 pre-OS 环境和网络能力 |

## 七、MBR 与 GRUB

MBR 即 Master Boot Record，位于磁盘开头的特殊启动扇区，大小为 512 字节。它同时保存预启动代码和分区信息：

| 区域 | 大小 | 内容 |
| --- | --- | --- |
| `0x000` 到 `0x1bd` | 446 字节 | 启动代码和数据 |
| `0x1be` 到 `0x1fd` | 64 字节 | 4 个分区表项，每项 16 字节 |
| `0x1fe` 到 `0x1ff` | 2 字节 | 启动签名 `0x55aa` |

BIOS 读取 MBR 时会检查最后两个字节是否为 `0x55aa`。如果签名不存在，通常认为该介质不可启动。

MBR 分区表最多只有 4 个表项，因此传统分区模型中：

+ 主分区最多 4 个。
+ 主分区和扩展分区总数不能超过 4 个。
+ 扩展分区不能直接使用，必须再划分逻辑分区。
+ 通常只有一个活动分区用于引导。

GRUB 的磁盘引导过程也体现逐级加载：

| 阶段 | 作用 |
| --- | --- |
| stage1 | 位于 MBR 的小段代码，负责加载下一阶段 |
| stage1.5 | 识别文件系统，使 GRUB 能从文件系统读取后续文件 |
| stage2 | 显示启动菜单，加载内核映像和可选的 `initrd` |

BIOS 通常把硬盘主引导记录读入 `0x7c00`，检查 `0x7dfe` 处的启动签名，然后跳转到 `0x7c00` 执行。主引导程序再寻找活动分区，读取该分区的引导扇区，并继续交接控制权。

## 八、Linux 内核映像和用户态启动

传统 x86 Linux 课件脉络中，内核映像可以看作由引导代码、启动代码和压缩的内核主体组成。历史上常见：

+ `bootsect.S`：装载系统启动过程并设置启动参数。
+ `setup.S`：初始化系统和硬件，切换到保护模式。
+ `video.S`：初始化显示设备。
+ `head.S`：进入 32 位内核早期入口，解压内核并跳到 `start_kernel()`。

典型流程是：

1. `bootsect.S` 从 `0x7c00` 搬移自身，读取 setup 区，设置栈和磁盘参数。
2. 根据内核大小，把内核映像加载到低端或高端内存。
3. `setup.S` 检测物理内存布局，初始化部分硬件，打开 A20，建立 GDT/IDT，切换到保护模式。
4. `head.S` 初始化段寄存器和临时栈，清零 BSS，解压内核。
5. 解压后的内核建立临时页表和进程 0 的内核栈，跳转到 `start_kernel()`。
6. `start_kernel()` 初始化调度、内存、中断、时间、slab 等内核模块。

内核建立完成后，才进入用户层初始化。课件使用的是传统 SysV init 线索：

+ 内核启动第一个用户层程序 `/sbin/init`。
+ `init` 读取 `/etc/inittab`，确定运行等级。
+ 执行 `/etc/rc.d/rc.sysinit`，设置 `PATH`、网络、swap、`/proc` 等。
+ 根据运行级别执行 `rc0.d` 到 `rc6.d` 中的脚本。
+ 执行 `/etc/rc.d/rc.local` 中的用户自定义初始化。
+ 执行 `/bin/login`，进入登录状态。

> [!CAUTION]
> 现代 Linux 发行版常用 `systemd` 替代传统 SysV init，但边界没有变：内核先完成自身初始化，然后启动第一个用户态 init 进程，由它继续拉起用户空间服务。

## 九、易错点

+ 固件、Bootloader、内核不是同一层。固件负责最早期硬件发现和控制转移，Bootloader 负责准备内核运行环境，内核才是操作系统主体。
+ MBR 是整个磁盘的第一个扇区；Boot Sector 是某个分区的第一个扇区。
+ Bootloader 不负责长期管理资源，它只负责把系统带到“内核能接管”的状态。
+ MIPS 选择 `kseg1` 启动，是因为它不依赖 TLB 和 Cache，而不是因为它性能最好。
+ 启动慢通常不是某一个步骤造成的，而是固件初始化、设备探测、内核初始化、模块加载和用户态服务启动共同叠加的结果。
