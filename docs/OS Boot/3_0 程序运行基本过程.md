# 程序运行基本过程

## 一、从 `hello world` 看程序执行

一个看起来很简单的程序：

```c
#include <stdio.h>

int main() {
    printf("hello world!\n");
    return 0;
}
```

真正运行时并不是“CPU 直接执行这个源文件”。它至少经历：

1. 源代码被编译、链接成可执行文件。
2. 用户通过 shell 或图形界面请求执行该文件。
3. 操作系统创建进程，并为它建立地址空间和内核管理数据结构。
4. 可执行文件的代码、数据和入口信息被映射到进程结构中。
5. 调度器选择该进程运行，CPU 从程序入口处开始取指令。
6. 程序通过系统调用和异常机制请求内核服务。

这一页关注的是“程序如何从磁盘上的文件变成正在执行的进程”。

## 二、程序、进程与作业

| 概念 | 形态 | 重点 |
| --- | --- | --- |
| 程序 | 静态文件或代码集合 | 存放在磁盘上，本身不占用 CPU |
| 进程 | 程序的一次执行过程 | 拥有 PCB、地址空间、打开文件、寄存器现场等运行状态 |
| 作业 | 用户提交给系统的一项任务 | 通常包含程序、数据和操作说明 |

> [!NOTE]
> 程序是静态的，进程是动态的。同一个程序可以被多次执行，对应多个不同进程。

## 三、从 shell 到新进程

用户告诉 shell 执行某个可执行文件后，操作系统会完成一组控制动作：

1. **创建新的执行实体。** 内核为进程分配进程控制块 PCB 或类似结构，记录进程标识、状态、资源、地址空间和调度信息。
2. **检查可执行文件。** 内核找到目标文件，检查文件类型和头部信息，识别代码段、数据段、入口地址、权限和所需装载信息。
3. **建立进程地址空间。** 可执行文件被映射到进程的虚拟地址空间，栈、堆、共享库和内核辅助区域也会按规则建立。
4. **设置 CPU 上下文。** 内核设置程序计数器、栈指针、参数、环境变量和必要寄存器，使进程第一次被调度时能从入口处开始执行。
5. **进入就绪队列。** 进程等待调度器分配 CPU。

这个过程体现了操作系统的两个核心职责：一是把静态程序组织成可运行对象，二是让多个可运行对象安全地共享 CPU 和内存。

相关内容可继续看 [进程状态与控制](../进程与线程/4_1%20进程状态与控制.md)。

## 四、第一条指令与缺页异常

程序开始执行第一条指令时，并不要求所有代码和数据都已经在物理内存中。现代系统常用请求分页：

1. CPU 按虚拟地址取指或访存。
2. MMU 查页表，发现目标页不在内存或权限不满足。
3. 触发缺页异常，CPU 陷入内核。
4. 内核检查该地址是否属于进程合法地址空间。
5. 若合法，分配物理页框，并把对应代码或数据从磁盘读入内存。
6. 内核更新页表，必要时刷新或更新 TLB。
7. 返回用户态，重新执行触发缺页的指令。

> [!IMPORTANT]
> 缺页异常不一定是程序错误。若地址合法但页面尚未调入，缺页是请求分页系统的正常工作机制；若地址非法或权限错误，才会导致段错误等异常结果。

相关内容可继续看 [虚拟内存管理](../内存管理/3_4%20虚拟内存管理.md) 和 [请求式分页系统](../内存管理/3_5%20请求式分页系统.md)。

## 五、`printf` 如何到达屏幕

`printf("hello world!\n")` 也不是用户程序直接控制显示器。大致路径是：

1. C 库函数先在用户态完成格式化，把输出内容组织成字节序列。
2. C 库通过 `write` 等系统调用请求内核把数据写到标准输出。
3. CPU 执行系统调用指令，进入内核态。
4. 内核检查用户缓冲区地址是否合法，避免用户程序传入非法内存。
5. 内核根据文件描述符找到目标对象，标准输出可能连接到终端、管道、文件或伪终端。
6. 若目标是终端，数据可能被送到终端驱动、伪终端控制进程或窗口系统。
7. 窗口系统把字符转换成字形和像素，显示硬件再把帧缓冲中的像素输出到屏幕。

所以，`printf` 这类简单调用背后连接了文件描述符、系统调用、设备抽象、驱动和显示系统。

## 六、系统调用、中断和上下文

程序运行期间，用户态和内核态之间会频繁切换：

| 场景 | 进入内核的原因 | 典型结果 |
| --- | --- | --- |
| 系统调用 | 程序主动请求 OS 服务 | 文件读写、进程创建、内存映射 |
| 缺页异常 | 当前指令访问的页不在内存或权限不满足 | 调页后重试，或向进程报告错误 |
| 时钟中断 | 定时器异步触发 | 调度器可能切换到另一个进程 |
| I/O 中断 | 设备完成请求 | 唤醒等待 I/O 的进程 |

系统调用只表示 CPU 从用户态进入内核态处理请求，不必然意味着切换到另一个进程。进程上下文切换则是调度器保存当前进程现场，并恢复另一个进程现场，两者相关但不是同一个概念。

## 七、执行链条总结

可以把程序运行看成下面这条链：

```txt
源代码
  -> 编译/链接
  -> 可执行文件
  -> shell 请求执行
  -> 内核创建进程
  -> 建立地址空间和 CPU 上下文
  -> 调度执行
  -> 缺页、系统调用、中断等事件反复发生
  -> 程序退出，内核回收资源
```

这条链把 OS Boot 章节和后续章节连接起来：

+ 进程管理解释“谁在运行、何时运行”。
+ 内存管理解释“虚拟地址如何对应物理内存”。
+ 设备管理解释“输出和输入如何通过统一接口完成”。
+ 文件系统解释“程序、数据和设备如何以文件形式被访问”。

## 八、易错点

+ 可执行文件不是进程；进程是可执行文件被装入并获得运行状态后的结果。
+ `printf` 不是直接写显存，通常会经过 C 库、系统调用、内核对象和终端或窗口系统。
+ 缺页异常可以是正常调页，也可以是非法访问，关键看地址是否合法以及权限是否满足。
+ 用户态进入内核态是 mode switch，不必然发生进程上下文切换。
